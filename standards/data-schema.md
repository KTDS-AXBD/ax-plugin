# 데이터/스키마 관리 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 스키마 설계 원칙

### 명명 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| 테이블 | 복수형 snake_case | `discoveries`, `users`, `token_usages` |
| 컬럼 | snake_case | `created_at`, `user_id`, `display_name` |
| FK 컬럼 | `{참조테이블_단수}_id` | `user_id`, `discovery_id` |
| 인덱스 | `idx_{테이블}_{컬럼}` | `idx_discoveries_status` |
| 유니크 | `uniq_{테이블}_{컬럼}` | `uniq_users_email` |

### 컬럼 타입 기준

| 데이터 | 권장 타입 | 비고 |
|--------|-----------|------|
| ID (PK) | `text` (ULID/UUID) 또는 `integer` (auto-increment) | 프로젝트별 선택 |
| timestamp | `integer` (unixepoch) | 플랫폼 호환성 높음 |
| boolean | `integer` (0/1) | SQLite 호환 |
| JSON | `text` (JSON 직렬화) | ORM 자동 직렬화 활용 시 수동 parse 금지 |
| enum/status | `text` | 타입 안전성은 코드 레벨에서 보장 |

### 인덱스 전략

- FK 컬럼은 기본적으로 인덱스 생성
- 검색/필터에 빈번하게 사용되는 컬럼에 인덱스
- 복합 유니크 제약 활용 (데이터 정합성 보장)
- 불필요한 인덱스는 쓰기 성능 저하 → 실제 쿼리 패턴 기반으로 추가

## 2. 마이그레이션

### 순방향 원칙 (forward-only)

- 마이그레이션은 항상 순방향. 롤백 스크립트 작성 금지
- 수정이 필요하면 새 마이그레이션 생성
- 이미 적용된 마이그레이션 파일 수정 금지

### 작성 규칙

- 마이그레이션 1개 = 논리적 변경 1개 (여러 테이블 동시 변경 가능하되 하나의 목적)
- 파일명에 순번 포함 (ORM 자동 생성 활용)
- 데이터 마이그레이션과 스키마 마이그레이션 분리 권장

### 테스트 동기화

- **새 마이그레이션 추가 시 테스트 helper에도 반드시 반영**
- 테스트용 DB 초기화 스크립트와 마이그레이션 파일 동기화
- 누락 시 "no such table" 에러 발생 (프로젝트 Gotchas)

### 적용 절차

```
1. 스키마 코드 수정
2. 마이그레이션 파일 생성
3. 로컬 DB에 적용
4. 테스트 helper 동기화
5. 테스트 실행으로 검증
6. 프로덕션 적용 (배포 시)
```

## 3. 데이터 정합성

### 제약 조건

- FK 제약: 참조 무결성 보장. 가능하면 DB 레벨에서 강제
- NOT NULL: 명시적으로 nullable이 필요한 경우만 nullable 허용
- UNIQUE: 비즈니스 규칙상 유일해야 하는 필드에 적용
- DEFAULT: 합리적 기본값이 있는 필드에 설정

### 소프트 딜리트 vs 하드 딜리트

- 프로젝트 특성에 따라 선택 (표준에서 강제하지 않음)
- 소프트 딜리트 시 `deleted_at` 컬럼 사용
- 하드 딜리트 시 관련 데이터 cascade 규칙 명확히 정의

### 상태 전환

- 상태(status) 컬럼의 허용 값은 코드 레벨에서 정의
- 상태 전환 규칙은 검증 함수 경유 필수 (직접 DB UPDATE 금지)
- 허용된 전환만 가능하도록 validation 로직 구현

## 4. 백업/복구

### 기본 원칙

- 프로덕션 데이터는 정기 백업 필수
- 백업 주기와 보존 기간은 프로젝트별 정의
- 복구 절차를 문서화하고 주기적으로 복구 테스트 권장
- 마이그레이션 적용 전 프로덕션 백업 권장

## 5. 스키마 문서화

- 스키마 변경 시 관련 SPEC/설계 문서에 반영
- 테이블 간 관계도는 프로젝트 규모가 커질 때 별도 문서 작성 권장
- ORM 스키마 코드 자체가 1차 문서 역할 (별도 ERD는 보조)
