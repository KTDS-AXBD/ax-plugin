# 모니터링/관찰성 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 로깅

### 로그 레벨

| 레벨 | 용도 | 예시 |
|------|------|------|
| ERROR | 즉시 대응 필요한 오류 | DB 연결 실패, 외부 API 장애 |
| WARN | 주의가 필요한 상황 | 재시도 발생, 느린 쿼리, 폴백 사용 |
| INFO | 정상 운영 이벤트 | 배포 완료, 배치 작업 시작/종료, 사용자 로그인 |
| DEBUG | 개발 중 디버깅 용도 | 변수 값, 실행 흐름 추적 |

### 로그 포맷

- **Structured logging 권장**: JSON 형태로 구조화된 로그
- 최소 필드: `level`, `message`, `timestamp`
- 권장 필드: `context` (모듈명), `requestId` (요청 추적용)

```
{ "level": "ERROR", "message": "DB connection failed", "context": "db", "timestamp": "..." }
```

### 프로덕션 규칙

- DEBUG 로그는 프로덕션에서 비활성화
- 민감정보 마스킹 필수 (보안 표준 §4 참조)
- 로그에 request body 전체 포함 금지
- 에러 로그에는 충분한 컨텍스트 포함 (재현 가능하도록)

### 로그 금지 항목

- 시크릿, 토큰, API 키
- 비밀번호 (해시 포함)
- 개인정보 평문 (이메일은 마스킹)
- 대용량 데이터 (페이로드 전체 등)

## 2. 에러 추적

### 에러 심각도 분류

| 심각도 | 대응 시간 | 예시 |
|--------|-----------|------|
| Critical | 즉시 | 서비스 다운, 데이터 손실, 보안 침해 |
| High | 1시간 이내 | 핵심 기능 장애, 결제 오류 |
| Medium | 다음 세션 | 비핵심 기능 오류, 부분 장애 |
| Low | 백로그 | UI 깨짐, 경미한 버그, UX 개선 |

### 에러 수집 원칙

- **모든 unhandled error는 수집**: 누락 없이 추적
- **에러 그룹핑**: 동일 원인의 에러는 그룹화하여 중복 알림 방지
- **컨텍스트 포함**: 에러 발생 시 요청 정보, 사용자 정보(마스킹), 스택 트레이스 포함
- **에러 수집 도구**: Sentry, LogRocket 등 프로젝트별 선택

### 에러 대응 절차

1. **감지**: 에러 수집 도구 또는 로그 모니터링으로 감지
2. **분류**: 심각도 판단 (위 표 기준)
3. **대응**: 심각도별 시간 내 대응 시작
4. **기록**: 원인과 해결 방법 기록 (재발 방지)

## 3. 헬스체크

### 기본 원칙

- 프로덕션 서비스는 헬스체크 엔드포인트 제공 권장
- 헬스체크는 핵심 의존성(DB, 외부 API 등) 상태 포함
- 배포 후 헬스체크 자동 확인 권장

### 헬스체크 응답

```json
{ "status": "ok", "timestamp": "..." }
```

- 정상: `200 OK`
- 이상: `503 Service Unavailable` + 상세 원인

## 4. 알림 원칙

- **Critical/High 에러만 즉시 알림** (알림 피로 방지)
- Medium 이하는 대시보드/로그에서 확인
- 알림 채널은 프로젝트별 선택 (Slack, 이메일 등)
- 동일 에러의 반복 알림 제한 (throttling)

|------|------|
| GOV-007 보안 | §4 데이터 보호 — 로그 마스킹 규칙, 민감정보 분류 |
| GOV-009 CI/CD | §3 배포 전략 — 배포 후 헬스체크 확인 |
| GOV-012 성능 | §2 API 성능 — 응답 시간 기준(p50/p95)과 모니터링 연계 |
