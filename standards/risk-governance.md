# 리스크 관리 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 리스크 유형

| 유형 | 정의 | 예시 |
|------|------|------|
| Blocker | 현재 작업을 진행할 수 없게 하는 장애물 | API 키 만료, 외부 서비스 장애 |
| Dependency | 다른 작업/시스템에 의존하는 항목 | 마이그레이션 선행 필요, 패키지 업데이트 대기 |
| Tech Debt | 작동하지만 개선이 필요한 코드/구조 | 하드코딩된 값, 중복 로직, 미흡한 에러 처리 |
| Constraint | 플랫폼/인프라 제약 | Cloudflare 30초 타임아웃, D1 SQLite 한계 |

## 2. 식별

### 식별 시점

| 시점 | 방법 |
|------|------|
| 세션 시작 | MEMORY.md "다음 작업" 점검 시 블로커 확인 |
| 구현 중 | 예상치 못한 제약/의존성 발견 시 즉시 기록 |
| 세션 종료 | `/ax-session-end` 시 미해결 블로커를 MEMORY.md에 기록 |
| 마일스톤 회고 | 기술부채 일괄 점검 |

### AI Agent 자동 감지

- 세션 중 반복되는 우회 패턴 → Tech Debt 후보 자동 플래깅
- 빌드/테스트 실패 패턴 → Blocker 후보 식별

## 3. 기록

### 기록 위치

| 위치 | 대상 |
|------|------|
| MEMORY.md "다음 작업" | 활성 Blocker — `[긴급]` 태그 |
| SPEC.md "기술 제약" | Constraint — 프로젝트 수명 동안 유효 |
| 요구사항 (Improvement) | Tech Debt — 정규 요구사항으로 등록 |

### 표기 규칙

MEMORY.md에서 리스크 항목은 다음 접두사를 사용:
```
- **[긴급]** API 크레딧 만료 — 영향: Agent 전체 비활성화
- **[블로커]** 마이그레이션 48 선행 필요 — 영향: 신규 테이블 사용 불가
- **[부채]** callLLM 에러 핸들링 통일 필요 — 영향: 장애 시 디버깅 어려움
```

## 4. 대응

### 에스컬레이션 규칙

| 수준 | 조건 | 대응 |
|------|------|------|
| 즉시 | 프로덕션 장애, 데이터 손실 위험 | 현재 작업 중단 + 즉시 수정 + 핫픽스 배포 |
| 세션 내 | 현재 작업의 블로커 | 우회 방법 모색 → 불가 시 작업 전환 |
| 다음 세션 | 현재 작업에 영향 없음 | MEMORY.md 기록 + 다음 세션 우선순위 반영 |
| 백로그 | 기술부채/개선사항 | 요구사항으로 등록 (Improvement/P2~P3) |

### 블로커 해소 원칙

1. **우회보다 해결 우선** — 근본 원인 파악 후 수정. 임시 우회는 Tech Debt으로 등록
2. **의존성은 미리 확인** — 세션 시작 시 필요한 외부 서비스/키/권한 점검
3. **동일 블로커 반복 금지** — 2회 이상 같은 블로커 발생 시 구조적 해결 (자동화/문서화)

## 5. 기술부채 관리

### 등록 기준

다음 중 하나에 해당하면 Tech Debt로 등록:
- 같은 우회 패턴이 2회 이상 반복
- "나중에 고치자"로 남긴 TODO가 2세션 이상 미처리
- 코드 리뷰/분석에서 개선 권고 발생

### 처리 원칙

- 마일스톤당 Tech Debt 1~2건 이상 처리 권장
- P3라도 3개 마일스톤 이상 미처리 시 P2로 자동 승격 검토
- 해소 시 요구사항 상태를 DONE으로 전환 + CHANGELOG 기록

### 해소 추적 (필수)

Tech Debt 해소 시 다음 정보를 반드시 기록한다:

| 항목 | 필수 | 형식 | 예시 |
|------|:----:|------|------|
| 해소 세션 | ✅ | `(세션 NNN)` | `해소 (세션 131)` |
| 해소일 | ✅ | YYYY-MM-DD | 등록일과 동일 컬럼 or 별도 |
| 표기 | ✅ | `~~TD-NN~~` 취소선 | `~~TD-02~~` |

**SPEC.md 기술부채 테이블 규칙:**
- 해소된 TD는 ID에 `~~취소선~~` 적용
- 영향 컬럼에 `해소 (세션 NNN)` 형식으로 세션 번호 기록
- 등록일 컬럼은 유지 (등록-해소 간격 추적용)
- 활성 TD(TD-01, TD-10 등)는 취소선 없이 유지

## 6. SPEC.md §8 "활성 기술부채" 테이블 형식

> GOV-005(리스크 관리) ↔ GOV-004(프로젝트 관리) 연동 규격

SPEC.md §8에 기술부채를 기록할 때 아래 테이블 형식을 사용한다:

```markdown
| ID | 등록일 | 제목 | 영향 | 상태 |
|----|--------|------|------|------|
| TD-01 | 2026-01-15 | cost 집계 미구현 | svc-governance 비용 대시보드 불가 | 활성 |
| ~~TD-02~~ | 2026-01-20 | Neo4j backfill 누락 | 해소 (세션 131) | 해소 |
```

### 컬럼 정의

| 컬럼 | 필수 | 설명 |
|------|:----:|------|
| ID | ✅ | `TD-NN` 형식. 해소 시 `~~TD-NN~~` 취소선 적용 |
| 등록일 | ✅ | YYYY-MM-DD. **삭제 금지** — 등록-해소 간격 추적에 필요 |
| 제목 | ✅ | 기술부채 요약 (1줄) |
| 영향 | ✅ | 활성: 영향 범위 설명 / 해소: `해소 (세션 NNN)` |
| 상태 | ✅ | `활성` 또는 `해소` |

### 규칙

1. **활성 TD**: 취소선 없이 유지, 상태 = `활성`
2. **해소 TD**: ID에 `~~TD-NN~~` 취소선 + 영향 컬럼에 `해소 (세션 NNN)` + 상태 = `해소`
3. **등록일 보존**: 등록일 컬럼은 절대 삭제하지 않음 (등록→해소 간격 측정용)
4. **MEMORY.md 동기화**: 활성 TD는 MEMORY.md "활성 리스크"에도 반영
