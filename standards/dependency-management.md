# 의존성 관리 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 업데이트 전략

### 버전별 분리

| 변경 수준 | 시점 | 검증 |
|-----------|------|------|
| patch | 즉시 적용 | 기존 테스트 통과 확인 |
| minor | 세션 내 적용 | 기존 테스트 + 영향 범위 확인 |
| major | 마일스톤 사이 계획 | Breaking Changes 목록 확인 + 마이그레이션 |
| 보안 | 즉시 (우선순위 최상) | `audit fix` 후 테스트 통과 확인 |

### 업데이트 절차

1. 변경 로그(CHANGELOG) 확인
2. Breaking Changes 유무 확인 (major)
3. 로컬에서 업데이트 후 테스트 실행
4. 문제 없으면 lock 파일과 함께 커밋

## 2. 패키지 매니저

### 프로젝트 내 통일

- 프로젝트 단위로 패키지 매니저 1개 통일 (npm, pnpm, yarn 중 선택)
- `packageManager` 필드(package.json)로 버전 고정 권장
- 팀원 간 다른 매니저 사용 금지 (lock 파일 충돌 방지)

### lock 파일

- **lock 파일은 반드시 git에 커밋** (재현 가능한 빌드 보장)
- lock 파일 수동 편집 금지
- lock 파일 충돌 시 삭제 후 재생성보다 신중한 해결 우선

## 3. 의존성 추가/제거

### 최소 원칙

- 기존 의존성 또는 표준 라이브러리로 해결 가능하면 새 의존성 추가 지양
- 추가 전 확인 사항:
  - 번들 크기 영향
  - 유지보수 상태 (마지막 업데이트, 이슈 처리율)
  - 라이선스 호환성
  - 대안 존재 여부

### 사유 기록

- 새 의존성 추가 시 커밋 메시지에 사유 명시
- 예: `chore: add zod — API 입력 검증 스키마 기반 처리`

### devDependencies 분리

- 런타임 불필요 패키지는 반드시 `devDependencies`에 설치
- 빌드 도구, 테스트 프레임워크, 린터, 타입 정의 등

## 4. 보안 감사

### 정기 감사

- `npm audit` / `pnpm audit` 정기 실행
- 보안 취약점 발견 시 즉시 대응 (보안 표준 참조)
- high/critical 취약점은 즉시 패치, moderate는 다음 세션 내 처리

### 미사용 의존성 정리

- 사용하지 않는 의존성은 발견 즉시 제거
- 주기적으로 `depcheck` 등으로 미사용 패키지 점검 권장

## 5. 버전 범위

### 권장 버전 범위

- **caret (`^`)**: 기본값. minor/patch 자동 업데이트 허용
- **exact**: 프레임워크 코어 등 안정성 중요한 패키지
- **tilde (`~`)**: patch만 자동 업데이트 (보수적)
- lock 파일이 실제 설치 버전을 고정하므로, 범위는 의도 표현용
