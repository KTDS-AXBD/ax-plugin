# CI/CD 파이프라인 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 빌드 파이프라인

### 4단계 순차 (Fail-fast)

```
typecheck → lint → test → build
   │         │       │       │
   └─ 빠름 ──┘       │       │
              └─ 보통 ─┘       │
                        └─ 느림 ─┘
```

| 단계 | 목적 | 실패 시 |
|------|------|---------|
| typecheck | 타입 안전성 검증 | 즉시 중단 |
| lint | 코드 품질 + 스타일 검증 | 즉시 중단 |
| test | 기능 정확성 검증 | 즉시 중단 |
| build | 프로덕션 빌드 생성 | 즉시 중단 |

### Fail-fast 원칙

- 빠른 검증부터 실행하여 피드백 루프 최소화
- 이전 단계 실패 시 이후 단계 스킵
- 각 단계의 실패 원인을 명확하게 출력

## 2. 브랜치 전략

### Trunk-based Development

```
master ──○──○──○──○──○──○──○─→
              └─○─○┘   (short-lived branch)
```

### 규칙

- **master가 항상 배포 가능 상태** 유지
- 소규모 변경은 master에 직접 push 허용
- 대규모 변경 또는 리뷰 필요 시 feature branch → PR 경유
- feature branch는 **1~2일 내 머지** (장기 브랜치 금지)
- 브랜치명: `{type}/{description}` (예: `feat/ontology-view`, `fix/session-timeout`)

### 커밋 규칙

- Conventional Commits: `feat:`, `fix:`, `docs:`, `refactor:`, `style:`, `chore:`
- 커밋 전 로컬 검증 (typecheck + lint + test) 통과 필수
- 의미 있는 단위로 커밋 (작업 1개 = 커밋 1개 지향)

## 3. 배포 전략

### push-to-deploy + 프리뷰

```
master push → CI 검증 → 프로덕션 자동 배포
                          └─ 실패 시 차단

프리뷰: 수동 트리거 또는 PR별 자동
```

### 프로덕션 배포

- master push 시 CI 검증 통과 후 자동 배포
- CI 실패 시 배포 차단 (수동 우회 금지)
- 배포 후 헬스체크 확인 권장

### 프리뷰 배포

- 기능 확인/리뷰 용도의 임시 배포
- 수동 트리거 (`/deploy --preview` 등) 또는 PR별 자동
- 프리뷰 URL은 일정 기간 후 자동 만료 또는 수동 정리

### 롤백

- **기본 방법**: `git revert` 후 재배포 (히스토리 보존)
- **긴급 시**: 플랫폼 롤백 기능 활용 (이전 빌드로 즉시 복원)
- `git reset --hard` + force push는 최후의 수단 (팀 합의 필요)

## 4. 환경 관리

### 2환경 (dev + prod)

| 환경 | 용도 | 데이터 |
|------|------|--------|
| dev (로컬) | 개발, 테스트 | 로컬 DB, 목 데이터 |
| prod | 프로덕션 서비스 | 실제 데이터 |

### 환경별 설정

- 환경변수로 환경 구분 (`NODE_ENV`, 플랫폼별 설정)
- 환경별 설정 파일 분리 (`.dev.vars`, wrangler.toml 등)
- 프로덕션 설정은 시크릿 관리자에서 관리 (보안 표준 참조)

### 환경 동기화

- 로컬 환경은 프로덕션과 최대한 동일한 구성 유지
- 로컬에서 재현 불가한 프로덕션 이슈는 프리뷰 환경에서 디버깅

## 5. CI 도구

### 권장 사항

- CI 도구 선택은 프로젝트/플랫폼에 따라 자유 (GitHub Actions, CF Pages 빌트인 등)
- 빌드 캐시 활용하여 CI 시간 단축
- CI 설정 파일도 코드 리뷰 대상에 포함
