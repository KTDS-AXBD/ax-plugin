# 성능 기준 표준

> 범용 표준 — 모든 프로젝트에 공통 적용

## 1. 웹 성능 예산

### 권장 범위

| 항목 | Good | Needs Work | Poor |
|------|------|------------|------|
| LCP (Largest Contentful Paint) | < 2.5s | 2.5~4s | > 4s |
| CLS (Cumulative Layout Shift) | < 0.1 | 0.1~0.25 | > 0.25 |
| INP (Interaction to Next Paint) | < 200ms | 200~500ms | > 500ms |
| 번들 크기 (gzip) | < 500KB | 500KB~1MB | > 1MB |
| 초기 로드 시간 | < 3s | 3~5s | > 5s |

- Google Core Web Vitals 기준 참조
- 프로젝트별 특성에 따라 조정 가능
- Edge 환경(Cloudflare 등)에서는 더 빠른 응답 가능

### 번들 관리

- 코드 스플리팅 활용 (라우트 기반 lazy loading)
- 트리 쉐이킹 가능한 패키지 선호
- 대용량 라이브러리 도입 전 번들 영향 확인 (`bundlephobia` 등)
- 이미지/미디어 최적화 (적절한 포맷, lazy loading)

## 2. API 성능

### 권장 응답 시간

| 대상 | Good | Needs Work | Poor |
|------|------|------------|------|
| API p50 | < 100ms | 100~300ms | > 300ms |
| API p95 | < 200ms | 200~500ms | > 500ms |
| DB 쿼리 | < 50ms | 50~200ms | > 200ms |
| 외부 API 호출 | < 1s | 1~3s | > 3s |

### API 성능 원칙

- N+1 쿼리 금지 — 관련 데이터는 JOIN 또는 배치 조회
- 불필요한 데이터 반환 금지 — 필요한 필드만 SELECT
- 페이지네이션 필수 — 대량 데이터 목록 API
- 캐싱 활용 — 변경 빈도 낮은 데이터에 적절한 캐시 전략

## 3. 최적화 원칙

### 측정 후 최적화

- **조기 최적화 금지**: 성능 문제가 확인되기 전에 최적화하지 않음
- **측정 기반**: 추측이 아닌 프로파일링/벤치마크 결과로 판단
- **병목 우선**: 가장 큰 병목부터 해결 (전체 성능에 영향이 큰 곳)

### 최적화 절차

```
1. 성능 문제 감지 (사용자 피드백, 메트릭 경고)
2. 프로파일링으로 병목 지점 식별
3. 개선안 구현
4. 벤치마크로 개선 효과 확인
5. 회귀 방지 (테스트 또는 성능 모니터링)
```

### 일반 최적화 패턴

- **Lazy loading**: 초기 로드에 불필요한 리소스는 지연 로딩
- **Debounce/Throttle**: 빈번한 이벤트(검색, 스크롤) 제어
- **메모이제이션**: 비용이 큰 계산 결과 캐싱 (남용 주의)
- **비동기 처리**: 시간이 오래 걸리는 작업은 비동기/백그라운드 처리

### 금지 패턴

- 동기 블로킹 호출 (UI 스레드 또는 서버 메인 스레드)
- 무한 재렌더링 유발 코드
- 인덱스 없는 대량 테이블 풀스캔
- 루프 내 개별 DB/API 호출 (N+1)
